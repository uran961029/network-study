## 01. 네트워크

> [!summary] 한줄 요약 네트워크란 노드들이 특정 장치나 규칙으로 연결된 유기적 시스템이며, 토폴로지·구성 요소의 조합으로 다양한 형태로 구현된다.

### 핵심 개념

#### 네트워크의 의미

- `네트워크` : 어떤 대상과 다른 대상을 특정한 장치나 규칙으로 연결한 유기적 시스템
    - 기술적 관점에서는 컴퓨터·스마트폰 등 기기나 시스템이 연결 대상 (예: 인터넷, 휴대전화망, 은행 ATM)
    - 소프트웨어 관점으로도 네트워크 개념은 적용됨 (AI 신경망, 디렉터리 트리 등)
- `노드` : 네트워크에서 연결되는 각각의 대상
    - 노드는 반드시 하나 이상의 다른 노드와 연결되어야 네트워크라 부를 수 있음
    - 분리된 단일 노드는 네트워크 개념에 해당하지 않음

#### 네트워크 토폴로지

- `네트워크 토폴로지` : 노드를 연결하는 방법이나 형태
    - **라인형** : 각 노드가 선형으로 연결
    - **링형** : 각 노드가 링(원형) 형태로 연결
    - **스타형** : 허브라는 기점 노드에 모든 노드가 연결
    - **버스형** : 모선 한 줄에 모든 노드가 연결
    - **트리형** : 각 노드가 부모 또는 자식 노드에 연결
        - `루트 노드` : 부모가 없는 최상위 노드
        - `리프(잎) 노드` : 자식이 없는 최하위 노드
    - **메시형** : 각 노드가 하나 이상의 노드와 임의로 연결
        - `풀 메시(풀 커넥트)` : 모든 노드가 모든 노드와 직접 연결된 형태
- **복합형 토폴로지** : 실제 대규모 네트워크는 단일 토폴로지가 아닌 복합형을 사용하는 것이 일반적
    - 예: 휴대전화망 → 기지국 기준 스타형 + 기지국 간 메시형/라인형 혼합
    - 물리적 연결 방법과 관리 방법이 반드시 일치하지는 않음 (인터넷 라우터는 메시형, 도메인 이름 체계는 트리형)

#### 네트워크 구성 요소 (LAN 기준)

- **LAN 케이블** : 기기 간 물리적 연결 매체
- `스위칭 허브(스위치)` : 네트워크 신호를 중계하는 장치. 여러 기기의 LAN 케이블을 연결하며, 지정된 노드에만 신호를 전송. 스위치로 연결된 범위가 네트워크 최소 단위
- `리피터` : 수신한 신호를 바로잡고 증폭하여 재전송. 접속 거리 확장 가능
- `브리지` : 프로토콜이나 전송 원리가 다른 네트워크 사이에서 신호를 중계. 신호 레벨·변조 방식·프로토콜 변환 수행
- `네트워크 인터페이스 카드(NIC)` : 네트워크 신호와 컴퓨터 내부 신호를 변환하는 회로 기판
- `라우터` : 스위치로 묶인 네트워크(세그먼트)끼리 연결하는 장치. LAN과 LAN을 연결하며, 수신 신호가 자신의 LAN으로 온 것인지 판단 후 내부 처리 또는 다른 라우터로 전달
- `게이트웨이` : 네트워크에 연결하는 접속점. 패킷 출입을 제어하며, 특정 기술을 지칭하는 용어가 아니라 라우터나 서버 등도 게이트웨이로 정의 가능
- `세그먼트` : 기기들이 하나의 허브로 연결된 네트워크의 최소 단위

## 02. 데이터 전송 원리

> [!summary] 한줄 요약 데이터는 직렬/병렬 전송과 동기/비동기 통신 방식으로 전달되며, 프로토콜이라는 약속 체계를 통해 이기종 기기 간 통신이 가능하다.

### 핵심 개념

#### 데이터 전송 방식

- `직렬 전송` : 하나의 신호선으로 펄스형 신호를 보내는 방식. 한 번에 1비트 전송
    - 현재 통신 네트워크 대부분이 직렬 전송 사용
    - 이론적으로 신호선 하나로 양방향 통신 가능하나, 일반적으로 송신용·수신용 각 1개씩 총 2개 사용
- `병렬 전송` : 여러 개의 신호선을 사용하여 한 번에 8비트 또는 16비트 단위로 데이터 전송
    - 여러 신호선이 필요하므로 장거리 연결에 부적합 → 네트워크보다는 컴퓨터 내부 등에 활용

#### 비동기 통신 vs 동기 통신

- `시작 비트` : 통신 시작을 알리는 신호
- `종료 비트` : 통신 종료를 알리는 신호
- `비동기 통신` : 시작 비트와 종료 비트를 사용하여 데이터의 시작/끝을 구분하는 방식
- `동기 통신` : **클럭**이라는 동기 신호를 사용하여 데이터를 보내는 방식
    - `클럭 신호` : 일정한 간격으로 보내는 기준 신호. 신호와 데이터는 클럭 신호의 펄스에 맞추어 전송됨

#### 프로토콜

- `프로토콜` : 네트워크에서 데이터를 주고받기 위한 통신 규약 및 약속
    
    - 데이터의 구성, 데이터를 확실하게 주고받는 절차 등이 정의됨
    - 같은 프로토콜을 사용하면 이기종 기기 간에도 통신 가능
    - 다른 프로토콜을 사용하면 같은 기기끼리도 통신 불가
    - 보통 표준화 단체나 컨소시엄에서 논의·검증을 거쳐 결정
      
- **주요 프로토콜 목록**

|프로토콜|전체 명칭|설명|
|---|---|---|
|**IEEE 802.3**|-|IEEE 규정 이더넷 규격. LAN 케이블 등에 사용|
|**IEEE 802.11**|-|IEEE 규정 무선 LAN 규격. Wi-Fi로 보급|
|**IP**|Internet Protocol|여러 네트워크를 연결하는 프로토콜. 인터넷 구성|
|**ARP**|Address Resolution Protocol|IP 주소에서 MAC 주소를 알아내는 프로토콜|
|**TCP**|Transmission Control Protocol|신뢰성을 높여 통신하는 프로토콜|
|**UDP**|User Datagram Protocol|속도와 편의성을 우선하는 프로토콜|
|**POP**|Post Office Protocol|이메일 수신 프로토콜|
|**SMTP**|Simple Mail Transfer Protocol|이메일 송신 프로토콜|
|**HTTP**|HyperText Transfer Protocol|웹 브라우저↔웹 서버 간 통신 프로토콜|
|**DNS**|Domain Name System|도메인 이름을 IP 주소로 변환하는 프로토콜|

## 03. 네트워크 계층

> [!summary] 한줄 요약 네트워크는 역할별로 계층을 나누어 설계하며, OSI 7계층 모델과 실제 인터넷의 TCP/IP 4계층 모델이 대표적인 프레임워크다.

### 핵심 개념

#### 계층 분리의 이유

- 애플리케이션이 프로토콜·OS·하드웨어 변환을 모두 처리하면 프로그램이 복잡해지고 낭비 발생
- 따라서 네트워크를 **하드웨어 / 프로토콜 / 애플리케이션** 레이어로 분담하여 처리

#### 프로토콜 스택

- `프로토콜 스택` : 다양한 프로토콜의 계층 구조
- 프로토콜의 구성:
    - `헤더` : 프로토콜 정보 (우편의 주소 정보에 해당)
    - `페이로드` : 데이터 본체
- 동작 원리: 하위 계층 프로토콜의 **페이로드**에 상위 계층의 헤더+페이로드를 포함시킴
    - 각 계층은 자신의 헤더 정보만 해석 후 페이로드를 상위 계층으로 전달
    - 이 과정을 **캡슐화(송신)** / **역캡슐화(수신)** 라고 함

![[Pasted image 20260503144359.png|476]]

#### OSI 참조 모델

- `OSI 모델` : ISO(국제표준화기구)가 1977년 제정한 7계층 네트워크 통신 모델
    - 현재는 OSI 모델을 그대로 구현한 네트워크나 장비는 거의 없음 (현재 기술의 기반이 된 모델이며, 이론적 참조 기준으로 사용)
    - **애플리케이션은 연결되는 컴퓨터나 케이블의 차이를 처리할 필요가 없음** → 계층 분리의 핵심 이점
    - 시스템을 프로토콜 스택 개념으로 설계·개발 가능

#### TCP/IP 모델

- `TCP/IP 모델` : 인터넷을 기준으로 한 4계층 네트워크 통신 모델 (현재 실제 사용되는 모델)

![[Pasted image 20260503145127.png|339]]

#### TCP/UDP/IP 프로토콜

- **IP 프로토콜**
    - UDP, TCP, HTTP 등 다른 프로토콜의 기반
    - IP 주소를 참조하여 페이로드를 올바른 목적지에 전송하는 최소한의 절차와 규칙 규정
    - 페이로드에는 UDP/TCP뿐 아니라 상위 계층 프로토콜 데이터도 저장 가능
      
- **UDP 프로토콜**
    - 헤더에 포트 번호 부여. 처리 과정은 IP와 거의 동일
    - 오류 검사·확인·재전송 처리 미수행 → 그만큼 오버헤드가 적고 빠름
    - `데이터그램` : 목적지만 단순히 지정하여 전송하는 데이터 단위
    - **UDP를 쓰는 경우** : 속도·실시간성이 중요하고, 일부 데이터 손실을 허용할 수 있을 때
        - DNS 쿼리 (요청-응답이 단순하고 빠를수록 좋음)
        - 영상 스트리밍 / VoIP (프레임 몇 개 빠져도 재생 가능, 재전송 기다리면 끊김)
        - 온라인 게임 (낮은 지연이 정확성보다 중요)
          
- **TCP 프로토콜**
    - 헤더에 포트 번호 부여
    - 오류 검사·확인·재전송 처리 수행 → 데이터가 반드시 도착함을 보장
    - 세션 통신 기능 포함 (3-way handshake로 연결 수립)
    - 헤더에 **시퀀스 번호, 응답 확인 번호** 등 추가 정보 포함
    - **TCP를 쓰는 경우** : 데이터 무결성이 중요하고, 손실이 허용되지 않을 때
        - HTTP/HTTPS (웹 페이지가 잘려서 오면 안 됨)
        - SSH (명령어 하나하나가 정확히 전달되어야 함)
        - DB 연결 — MySQL(3306), PostgreSQL(5432) 등 (데이터 누락은 치명적)
        - 파일 전송 FTP (파일이 깨지면 안 됨)

> TCP vs UDP 판단 기준 **"데이터가 하나라도 빠지면 안 되는가?"** → Yes면 TCP, No면 UDP 검토

## 04. IP 주소

> [!summary] 한줄 요약 IP 주소는 네트워크상의 출발지·목적지를 식별하는 주소로, 인터넷용 글로벌 IP와 내부용 프라이빗 IP로 구분되며 현재 IPv4에서 IPv6로 전환이 진행 중이다.

### 핵심 개념

#### IP 주소 기본

- `IP 주소` : TCP/IP 프로토콜에서 출발지나 목적지를 식별하는 식별자
- 크게 **IPv4**와 **IPv6** 두 가지 버전이 있음

#### IPv4

- **32비트**로 구성된 주소 체계
- 32비트를 **8비트씩 4등분**하여 표기 (점으로 구분된 10진수, 예: 192.168.1.1)
    - `옥텟` : 네트워크에서 8비트 단위를 부르는 명칭
- 약 **43억 개**(2³²)의 주소 표현 가능
    - 단, 예약된 특수 주소 공간 및 프라이빗 IP 주소 공간은 인터넷에서 사용 불가

#### 글로벌 IP 주소 vs 프라이빗 IP 주소

- `글로벌 IP 주소` : 인터넷 주소로 사용 가능한 IP 주소. 인터넷상에 하나의 기기만 가질 수 있는 고유 주소
    
- `프라이빗 IP 주소` : 인터넷에 직접 연결되지 않고 LAN 내부에서 사용하는 IP 주소
    
    - 인터넷에서 라우팅되지 않으므로 LAN마다 중복되지 않게만 관리하면 다른 LAN에서 같은 주소 사용 가능
- **프라이빗 IP 대역 및 규모 비교**
    

|대역|CIDR|사용 가능 호스트 수|주요 용도|
|---|---|---|---|
|10.0.0.0|/8|약 **1,677만 개**|클라우드 VPC, 대규모 기업망|
|172.16.0.0|/12|약 **104만 개**|중간 규모 기업망|
|192.168.0.0|/16|약 **6만 5천 개**|가정용 공유기, 소규모 LAN|

> [!tip] 클라우드 VPC는 왜 10.x 대역을 쓸까? VPC 내부에서 서브넷을 여러 개로 나눠야 하므로 주소 공간이 넉넉한 10.0.0.0/8을 주로 사용. 예: 10.0.0.0/16으로 VPC를 만들면 내부에서 /24 서브넷을 256개까지 쪼갤 수 있음.

#### IPv6

- IPv4 주소 고갈 문제를 해결하기 위해 책정된 새로운 주소 체계
- **128비트** (16옥텟)으로 IP 주소 표현 → 사실상 무한에 가까운 주소 공간
- IPv6의 장점:
    - 프로토콜 수준에서 **암호화 통신** 규정
    - **MAC 주소를 이용**하여 IPv6 주소 자동 생성 가능 (2계층 포함 설정 자동화)
    - 헤더가 IPv4보다 **단순**하여 처리 효율 향상
    - **계층 구조가 엄격**하여 라우팅 효율 우수
- IPv6의 제약:
    - IPv4와 **프로토콜 호환성 없음** → IPv6로 인터넷 연결 시 IPv6 호환 앱·서비스 필요

## 05. 서브넷

> [!summary] 한줄 요약 서브넷은 하나의 IP 네트워크를 CIDR 방식으로 분할한 세그먼트이며, 서브넷 마스크(프리픽스)로 네트워크부와 호스트부를 구분한다.

### 핵심 개념

#### 네트워크 분할 방식

- 일반적인 LAN은 장비의 종류·용도·사용 부서 등에 따라 세그먼트로 분할됨
- **클래스 방식** (과거): IP 주소 상위 **1~4비트** 패턴에 따라 A~E 클래스로 구분
    - A클래스(0xxx): 대규모 네트워크
    - B클래스(10xx): 중규모 네트워크
    - C클래스(110x): 소규모 네트워크
    - D클래스(1110): 멀티캐스트용 예약
    - E클래스(1111): 실험용 예약 (실제 사용 안 함)
    - 클래스 단위로만 나뉘어 낭비가 심해 현재는 거의 사용하지 않음
- `CIDR(Classless Inter-Domain Routing)` : IP 주소의 임의 비트 위치에서 네트워크부와 호스트부를 구분하는 방식 → 현재 표준

#### 네트워크부 vs 호스트부

- `네트워크부` : IP 주소 상위에서 임의의 비트 수를 사용하여 해당 네트워크를 대표하는 주소 범위
- `호스트부` : 해당 네트워크 내에서 사용 가능한 개별 주소 범위

#### 서브넷과 서브넷 마스크

- `서브넷` : CIDR 방식으로 분할한 네트워크 단위
    
- `서브넷 마스크` : 네트워크부와 호스트부를 구분하는 표기법
    
    - 네트워크부의 비트는 모두 **1** (10진수 255), 나머지는 모두 **0**으로 표기
    - 예: /24 → 255.255.255.0 (상위 24비트가 네트워크부)
- `프리픽스 표기` : IP 주소 뒤에 슬래시와 네트워크부의 비트 수를 붙인 표기
    
    - 예: `192.168.100.20/24` → 상위 24비트가 네트워크부, 하위 8비트가 호스트부
    - 개별 호스트의 IP 주소만으로는 서브넷 구분 비트 수를 알 수 없으므로 프리픽스 표기 필수
- **자주 쓰는 프리픽스 빠른 참조표**
    

|프리픽스|서브넷 마스크|호스트 수 (실사용)|클라우드 용도 예시|
|---|---|---|---|
|/16|255.255.0.0|65,534|VPC 전체 대역|
|/24|255.255.255.0|254|일반 서브넷|
|/28|255.255.255.240|14|소규모 서브넷, NAT GW용|
|/32|255.255.255.255|1|단일 호스트 지정, 보안 그룹 룰|

> [!note] 호스트 수 계산법 2^(호스트부 비트 수) - 2 = 실사용 가능 호스트 수 (-2는 네트워크 주소와 브로드캐스트 주소를 제외한 것) 예: /24 → 2^8 - 2 = **254개**

### 클라우드 연결 포인트

- **VPC 서브넷 설계**: 클라우드에서 VPC 내 서브넷을 생성할 때 CIDR 표기(/24, /28 등)를 직접 입력 가능. 
- **퍼블릭 서브넷 vs 프라이빗 서브넷 분리**: 인터넷 게이트웨이와 연결된 퍼블릭 서브넷과 내부 통신 전용 프라이빗 서브넷을 CIDR로 분리하여 보안 경계를 설계
- **CIDR 겹침 방지**: VPC 간 피어링이나 VPN 연결 시 CIDR 대역이 겹치면 라우팅 불가. 설계 단계에서 전사 IP 대역 관리 정책 수립이 필수

## 06. 포트 번호

> [!summary] 한줄 요약 포트 번호는 TCP/UDP 헤더에 포함된 16비트 식별자로, IP 주소가 지정한 서버에서 어떤 애플리케이션으로 패킷을 전달할지를 구분하는 역할을 한다.

### 핵심 개념

#### 포트 번호의 역할

- `포트 번호` : TCP/UDP 헤더의 처음 16비트 정보 (0~65535 범위)
    - IP 주소는 목적지 **서버**를 지정할 수 있지만, 해당 서버의 어떤 **앱**을 사용할지는 지정 불가
    - `목적지 포트 번호` : 패킷이 어떤 애플리케이션을 위한 것인지 나타내는 번호. 서버는 수신 패킷이 웹 요청인지 이메일 요청인지 등을 이 번호로 판단
    - `출발지 포트 번호` : 패킷을 보낸 쪽 사용자와 터미널을 식별하는 번호

#### 포트 번호 범위

|범위|구분|설명|
|---|---|---|
|**0~1023**|Well-known Port (잘 알려진 포트)|주요 애플리케이션에 사전 할당. 관례적 표준|
|**1024~49151**|Registered Port (등록된 포트)|기존 애플리케이션에서 보조적으로 사용|
|**49152~65535**|Dynamic / Private Port|개인·기업이 자체 애플리케이션용으로 자유롭게 사용 가능|

- 주요 Well-known 포트 예시:
    - **80**: HTTP / **443**: HTTPS / **22**: SSH / **3306**: MySQL / **5432**: PostgreSQL

#### 포트 번호의 필요성

- LAN 내 말단 컴퓨터는 **프라이빗 IP 주소**를 사용하며, 인터넷에서는 라우터·서버 등 LAN 구성 장치까지만 특정 가능
- 따라서 포트 번호로 LAN 내 장치나 단말기, 애플리케이션 종류까지 지정해야 통신 가능

## 07. MAC 주소

> [!summary] 한줄 요약 MAC 주소는 네트워크에 연결된 모든 기기가 고유하게 가지는 48비트 하드웨어 식별자이며, ARP 프로토콜을 통해 IP 주소와 매핑되어 LAN 내 통신에 사용된다.

### 핵심 개념

#### MAC 주소의 구조

- `MAC 주소(Media Access Control Address)` : 네트워크에 연결되는 모든 기기에 하나씩 부여되는 고유 식별자
    - 컴퓨터의 NIC, 스마트폰, 라우터, 스위치 등 모든 네트워크 연결 기기에 부여
    - **48비트** 길이의 식별자
    - `벤더 코드(OUI, Organizationally Unique Identifier)` : 상위 24비트. IEEE가 장비 제조사에 할당. 제조사는 여러 개의 벤더 코드 보유 가능
    - `노드 번호` : 하위 24비트. 각 제조사의 책임 하에 제품마다 하나씩 할당
    - IEEE(미국전기전자학회)가 전 세계 기기 제조사에 고유 코드를 관리·할당

#### MAC 주소 vs IP 주소

|구분|MAC 주소|IP 주소|
|---|---|---|
|**계층**|OSI 2계층(이하)|OSI 3계층 이상|
|**범위**|LAN 내 장치·단말기 식별|네트워크·장치 식별 (인터넷 포함)|
|**할당**|제조 시 하드웨어에 고정|네트워크 관리자 또는 DHCP가 동적 할당|
|**프로토콜**|이더넷 프레임|TCP/IP|

#### ARP (Address Resolution Protocol)

- `ARP` : IP 주소에서 해당 장비의 MAC 주소를 확인하는 프로토콜
- **필요한 이유**: 애플리케이션은 IP 주소를 알아도 MAC 주소는 모름. LAN 내에서 패킷을 교환하려면 IP→MAC 매핑이 필요
- **동작 방식**:
    1. `브로드캐스트` : ARP 요청 패킷을 대상 지정 없이 LAN 전체에 전송 (모든 장치에 수신됨)
    2. `유니캐스트` : 해당 IP를 가진 장치가 자신의 MAC 주소를 1:1로 회신
    3. `ARP 캐시(주소 테이블)` : 매번 ARP로 조회하는 것은 비효율적이므로, **각 호스트(단말기)** 에 IP↔MAC 대응표를 저장하여 활용
        - 스위치는 별도로 **MAC 주소 테이블**(포트↔MAC 대응)을 관리하며 이는 ARP 캐시와 다른 개념
        - ARP 캐시는 운영체제 수준에서 관리되며, 일정 시간이 지나면 만료(TTL)되어 갱신됨

> [!note] ARP 캐시 확인 명령어
> 
> - Windows / Linux: `arp -a`
> - 클라우드 환경에서는 VPC 내 통신도 동일한 ARP 원리로 동작하며, 클라우드 제공자가 가상 ARP를 처리함
